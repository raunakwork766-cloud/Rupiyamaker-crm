🛠️ Developer Notes
Core CRM logical rules & behaviour reference
1Time Input Logic
Typing '1' in the time field restricts the dropdown to exactly 1:00 AM / 1:00 PM and will not jump to 11:00. The filter uses an exact hour-part match — parseInt(query) === parseInt(hourPart). This prevents amiguous disambiguation when typing single digits.
2Record Search (Naming / Number Lookup)
The record selector dropdown supports native inline search — type any part of the customer name or mobile number and the list filters in real-time. There is no dependency on a separate search endpoint; all filtering happens client-side via data-name and data-number attributes on each list item.
3Remark History Structure
Every action (Created, Completed, Failed, Reopened) mandatorily stores a remark before proceeding. To-Do tasks use a full Description field instead of a short subject — the Description acts as the primary detail node in history. Callback & Pendency use a short subject/note. Each history entry contains: status, note (remark), taskNote (original task detail), user, and dateTime.
4Schedule Next — Queue Behaviour
Clicking "Schedule Next ➡️" on a Callback or Pendency task automatically marks the current task as Completed (with note "Next phase scheduled"), then opens a fresh Create form. The new task inherits the full history of the parent so the complete lifecycle is preserved in one chain. The _prevTaskIdForNext flag tracks the chain reference.
